# Second Brain for Home Assistant Voice Assistant

## Context

The `local_openai` component (this repo) connects local LLM backends to HA's voice pipeline, but the Assist pipeline is limited by design: no historic data, no persistent knowledge about the home, system prompt is static. Goal: a full personal assistant with a "second brain" — persistent, markdown-based knowledge (Karpathy LLM-Wiki style) the assistant reads every turn and writes to when told "remember X", plus deeper HA access (history, automations) via ha-mcp.

Research (last30days, 2026-07): community converged on plain-markdown folders as agent memory (Karpathy LLM Wiki, Google OKF format); MCP is the standard access layer; consensus warns AI writes need an audit mechanism. Decisions made with user:

- **One markdown store, two frontends, day one**: HA `llm.API` for any conversation agent (fast voice path: standing context + tools) AND MCP access for external clients (Claude Desktop/Code).
- **Store**: plain markdown + minimal YAML frontmatter, Obsidian-compatible, in `/config/second_brain/`, no embeddings in v1.
- **Writes**: self-learning — assistant writes immediately via `remember` tool; every write is a git commit authored as the assistant; review is post-hoc via git log/diff (review panel later).
- **This repo (`local_openai`) is NOT modified.** New standalone HACS repo; works with any HA conversation agent.

Key design findings (verified):

- HA core's **`mcp_server` integration exposes any registered `llm.API`** over MCP (streamable HTTP at `/api/mcp`, long-lived-token auth; older cores: SSE at `/mcp_server/sse`). So external MCP access = enable core integration + select "Second Brain" — **zero custom MCP server code needed**.
- HA core's `mcp` **client** integration is SSE-only today — relevant for ha-mcp wiring only.
- `git` binary ships in the official HA container → plain `git` subprocess, no dulwich. Guard with `shutil.which("git")`, degrade gracefully (no commits) if absent.
- Zero manifest `requirements` — stdlib + HA helpers only.
- `local_openai` already supports multiple LLM APIs per agent (`CONF_LLM_HASS_API` list); users just tick "Second Brain" in agent options.
- Local-model constraints: keep tools tiny (3), agents cap ~10 tool iterations/turn.

## Store layout (`/config/second_brain/`)

```
CORE.md          # pinned context, user-edited; injected into every turn (seeded with template)
INDEX.md         # generated by component (setup + after every write); never hand-edited
memories/        # assistant-written: inbox.md or <topic-slug>.md; timestamped bullets
wiki/            # curated notes; user edits via Obsidian/Samba/git
.git/            # audit trail
```

- Frontmatter: `title`, `tags`, `created` — written on file creation, parsed with a ~15-line prefix parser (no YAML lib).
- Component owns INDEX.md (path, title, tags, mtime per file); assistant never maintains it. Search never depends on it (live scan), so manual edits are picked up lazily.

## Behavior

**Standing context (api_prompt)**: CORE.md clipped to ~4k chars + INDEX summary clipped to ~2k chars + 2 lines of tool guidance. Constants in `const.py`.

**Tools (3, short schemas)**:
- `search_brain(query)` → naive scored scan of all .md files (filename ×8, tag ×6, heading ×4, body ×1), top 5 with snippets. `# ponytail: O(files×lines) scan; inverted index only if store outgrows ~1k files`
- `read_note(path)` → full content clipped ~8k chars. **Path traversal guard required** (`resolve().is_relative_to(store_root)`) — MCP-exposed trust boundary.
- `remember(text, topic?)` → append timestamped bullet to `memories/<slug>.md` (create with frontmatter if new) → regenerate INDEX.md → `git add -A && git commit` with `--author "Second Brain Assistant <...>"` and `-c user.name/-c user.email` from config. Global asyncio.Lock serializes writes (`# ponytail: household-scale`).

**Registration**: `entry.async_on_unload(llm.async_register_api(hass, BrainAPI(hass, store)))`.

## MCP + ha-mcp (docs, not code)

README sections:
1. **External access**: add core `mcp_server` integration, select "Second Brain" → `claude mcp add --transport http second-brain https://<ha>/api/mcp -H "Authorization: Bearer <token>"` (SSE path on older cores).
2. **Deep HA access**: install ha-mcp via HACS → connect via core `mcp` client (SSE URL) → tools appear as selectable LLM API in local_openai → curate small subset (local models degrade past ~10 tools).

## Phasing

- **v1**: everything above (MCP external access came free via mcp_server).
- **v2**: options flow (budgets, git toggle), update/archive tool if usage demands.
- **v3**: review panel (recent assistant commits + one-click revert), richer ha-mcp docs.

## Verification

- **Unit**: `test_store.py` — tmp store: search ranking, remember append/frontmatter/INDEX regen, `git log -1` author+message. `test_llm_api.py` — config entry setup, `llm.async_get_api`, api_prompt contains CORE.md, each tool via `async_call`, `read_note("../../secrets.yaml")` rejected.
- **Manual e2e**: install in dev HA → add integration → in local_openai agent options add "Second Brain" LLM API → voice "remember the boiler service is due in October" → check `git -C /config/second_brain log` → new conversation "when is the boiler service due?" → confirm `search_brain` fires + correct answer.
- **MCP e2e**: enable `mcp_server`, connect Claude Code with long-lived token, repeat remember/recall.

## Risks

- `mcp_server` transport differs by HA version — doc both, verify on user's HA during e2e.
- Combined prompt (Assist entities + brain context + tools) may pressure small local models — char clips are the knob.
- Non-container HA installs may lack git — `which` guard, graceful degrade.
